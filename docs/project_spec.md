# Учебная практика

## 1. Название проекта
DocValidator

## 2. Цель проекта
Создать программу для автоматизированной проверки каталога документов по заданному перечню требований с формированием отчета.

## 3. Кто пользователь программы
Целевой пользователь программы (End-User) — системный администратор или специалист отдела документооборота, имеющий доступ к командной строке и обладающий правами чтения файлов в проверяемом каталоге, а также на запись в каталог для сохранения отчетов.

Программа не требует навыков программирования, но предполагает базовое понимание работы с файловой системой и ключами командной строки.

## 4. Какие входные данные принимает программа
Программа построена на основе CLI-интерфейса и принимает аргументы командной строки двумя способами:

- Обязательные аргументы:
  - путь к каталогу, в котором необходимо выполнить проверку
  - путь к JSON-файлу требований, в котором перечислены ожидаемые файлы, допустимые расширения и параметры проверки;

- Дополнительные (опциональные) ключи:
  - `--output <путь>` — задает файл для сохранения отчета. Если ключ не указан, отчет выводится в консоль
  - `--format <txt|json|csv|pdf>` — формат выходного файла. Если формат не указан, но запись идет в файл, по умолчанию используется `.txt` с кодировкой UTF-8
  - `--short` / `--full` — определяет уровень детализации отчета
  - `--debug` — включает вывод служебных сообщений для отладки

## 5. Какие выходные данные формирует программа
Программа формирует отчет о своей работе, а затем по изначальному указанию пользователя:

- выводит отчет на экран
- либо записывает его в файл одного из допустимых форматов: `.txt`, `.json`, `.csv`, `.pdf`

Длина отчета также зависит от пользовательского ввода.

## 6. Что программа должна проверять
Программа должна проверять существование целевого каталога, корректность файла требований, наличие обязательных файлов в каталоге, соответствие найденных файлов допустимым расширениям, наличие лишних файлов и наличие пустых файлов.

## 7. Какие ошибки программа должна находить
Программа должна находить следующие ошибки:

- отсутствие целевого каталога
- отсутствие переданных файлов для поиска
- отсутствие ожидаемых файлов в каталоге
- несоответствие фактических расширений с переданными
- наличие лишних файлов в каталоге
- некорректный пользовательский ввод
- ошибка записи отчета
- некорректная структура JSON-файла требований

## 8. Какие форматы отчета будут поддерживаться
`.txt`, `.json`, `.csv`. Дополнительно планируется реализация `.pdf`-отчета.

## 9. Какие ограничения есть у первой версии
- Проверка распространяется только на файлы с расширениями `.docx`, `.doc`, `.pdf`, `.xlsx`, `.csv`, `.txt`, `.json`. Файлы других форматов игнорируются и сопровождаются соответствующей записью в логе и соответствующим пользовательским выводом.
- Отчеты сохраняются только с кодировкой UTF-8, другие кодировки не поддерживаются.
- Обработка файлов выполняется последовательно (однопоточно). Многопоточность не реализована.
- Все сообщения и отчеты выводятся на русском. Локализация не предусмотрена.
- Автоматический запуск по расписанию не поддерживается. Предусмотрен только ручной запуск посредством командной строки.
- Графический интерфейс (GUI) отсутствует. Реализовано взаимодействие только через CLI.

## 10. По каким критериям считаем, что программа готова
Программа считается готовой, если она корректно обрабатывает валидные и невалидные входные данные, формирует отчёты в заявленных форматах, имеет автоматизированные тесты для основных сценариев и содержит инструкцию по запуску.

## 11. Что планируется добавить в дальнейшем
- Графический интерфейс для более удобного пользовательского опыта.
- Поддержка новых расширений для парсинга файлов.
- Разнести обработку файлов на разные ядра (многопоточность) с целью уменьшить время выполнения программы.
- Добавить поддержку английского языка для сообщений и отчетов.
