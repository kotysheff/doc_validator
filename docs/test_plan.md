## Контекст тестирования

### Цель тестирования
Убедиться, что программа DocValidator корректно проверяет каталог документов по входящему JSON-файлу требований, выявляет:
- отсутствующие файлы
- лишние файлы
- пустые файлы
- файлы с недопустимыми расширениями

и формирует отчеты по результатам проверки.

### Объект тестирования
CLI-приложение DocValidator, предназначенное для автоматизированной проверки каталогов документов и формирования отчетов о результатах проверки.

### Область тестирования
В область тестирования входит:
- обработка аргументов командной строки
- обработка входного JSON-файла с перечислением требований
- проверка существования целевого каталога
- сканирование файлов в каталоге
- проверка наличия обязательных файлов
- проверка допустимых расширений
- выявление пустых, лишних и отсутствующих файлов
- формирование отчетов в форматах `.txt`, `.json`, `.csv`

### Что не тестируется
- графический интерфейс, так как в первой версии реализован только CLI-интерфейс
- многопоточная обработка, так как в первой версии поиск работает последовательно
- выполнение по расписанию
- локализация на другие языки
- - содержимое документов, так как программа проверяет наличие файлов, расширения и базовые свойства, а не анализирует текст внутри документов

## Тестируемые функции

1. **CLI-интерфейс**
   - запуск с обязательными аргументами: путь к каталогу и путь к JSON-файлу требований
   - запуск с опциональными ключами: `--output`, `--format`, `--short`, `--full`, `--debug`
   - обработка некорректных и отсутствующих аргументов

2. **Загрузка требований**
   - чтение корректного JSON-файла
   - обработка отсутствующего JSON-файла
   - обработка некорректной структуры JSON

3. **Сканирование каталога**
   - обработка существующего каталога
   - обработка отсутствующего каталога
   - получение списка файлов и их свойств

4. **Валидация**
   - поиск отсутствующих обязательных файлов
   - выявление неверных расширений
   - выявление пустых файлов
   - выявление лишних файлов

5. **Формирование отчетов**
   - вывод в консоль
   - сохранение отчета в форматах `.txt`, `.json`, `.csv`
   - формирование PDF-отчета рассматривается как дополнительная функция после реализации основной версии

## Виды тестирования

- **Функциональное тестирование** — проверка соответствия поведения программы заявленным требованиям
- **Негативное тестирование** — проверка ошибочных ситуаций: неправильный путь, поврежденный JSON, отсутствие файла
- **Модульное тестирование** — pytest-тесты отдельных модулей
- **Интеграционное тестирование** — проверка последовательности работы и потока данных: CLI → loader → scanner → validator → report
- **Регрессионное тестирование** — повторная проверка отсутствия изменений в поведении программы после исправления багов

## Тестовая среда

- Операционная система: MacOS Tahoe 26.5.1 / Windows 10 22H2
- Язык программирования: Python 3.10+
- Среда разработки: PyCharm / Visual Studio Code
- Система контроля версий: Git
- Тестовый фреймворк: pytest

## Тестовые данные

Для тестирования будут подготовлены следующие наборы данных:
- `valid_documents` — каталог с обязательными файлами и корректными расширениями
- `missing_files` — каталог, в котором отсутствует один или несколько обязательных файлов
- `wrong_extensions` — каталог с файлами недопустимого расширения
- `empty_files` — каталог, содержащий пустой файл
- `extra_files` — каталог с лишними файлами, не указанными в JSON-файле требований
- `invalid_requirements` — набор некорректных JSON-требований

## Критерии начала тестирования

Тестирование начинается после реализации минимальной версии программы, включающей:
- парсинг пользовательского ввода
- чтение JSON-файла требований
- сканирование каталога
- формирование результата проверки

Для модульного тестирования допускается проверка отдельных модулей сразу после их реализации.

## Критерии завершения тестирования

Тестирование считается завершенным, если:
- пройдены все чек-листы и тест-кейсы
- критические ошибки отсутствуют
- программа корректно обрабатывает валидные и невалидные входные данные
- отчеты формируются в заявленных форматах
- все автоматизированные pytest-тесты проходят успешно

## Риски

- некорректная структура JSON-файла требований
- различия в путях Windows / Linux
- проблемы с кодировкой при сохранении отчетов
- ошибки доступа к файлам и каталогам
- неправильная обработка файлов с одинаковыми именами
- неоднозначность при наличии файлов с одинаковым именем, но разными расширениями
- некорректная обработка сложных имён файлов, например `report.final.docx`
